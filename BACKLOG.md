> 1 item = 1 session Claude (issue labellisée `claude` ou session Cloud) = 1 PR.
> Cocher + lien PR quand c'est mergé. `/dispatch` et `/backlog` (claude-ops) lisent ce fichier.
> Priorité optionnelle en tête d'item : `(P1)` urgent · `(P2)` important · `(P3)` un jour.

- [x] (P1) Le merge auto ne ferme pas l'issue — `dispatch.yml` merge sous `GH_TOKEN: ${{ github.token }}`, or GitHub n'exécute pas les actions en cascade déclenchées par le `GITHUB_TOKEN` par défaut (garde anti-boucle) : le `Closes #N` du corps de PR est donc **ignoré**, et l'issue reste ouverte après merge. Conséquence : l'anti-collision de `/dispatch` et `/backlog` voit des sessions fantômes sur tous les repos, et le Project « Flotte » ne passe jamais en « Mergé » seul. Constaté le 2026-07-17 sur notes-bac-visualisateur#21 et bac-maths#57 (restées ouvertes, mergées par `app/github-actions`) vs Recherche-Emploi#28 (fermée seule, PR mergée à la main avec le compte Thibaud888) — les 3 PR portaient pourtant `Closes #N`. Même racine que l'item `pr-ready` ci-dessous. Correctif le moins cher : `gh issue close` explicite après `gh pr merge` dans l'étape « Merge automatique si CI verte » (0 secret requis) ; alternative : `FLEET_GH_TOKEN` (PAT) au lieu du `github.token`, ce qui réglerait aussi `pr-ready` — à arbitrer ensemble. DoD : une issue dispatchée se ferme seule au merge de sa PR, constaté sur un vrai cas. ✅ 2026-07-17 — arbitrage : `gh issue close` explicite (0 secret) ; correctif appliqué en local, commit direct sur `main` (l'issue [#5](https://github.com/Thibaud888/fleet-kit/issues/5) l'a prouvé : une session Actions ne peut pas modifier `.github/workflows/*`, permission `workflows` absente du token d'app). Constat « vrai cas » à faire au prochain dispatch.
- [x] (P2) L'allowlist des sessions n'a aucune porte de sortie réseau — `dispatch.yml` autorise `Bash(git|gh issue|gh pr|npm|npx|node|python|pip|pytest)`, `Edit/Write/Read/Glob/Grep` : ni `curl`, ni `WebFetch`, ni `gh workflow run`, ni `gh run`. Conséquence : **aucune session dispatchée ne peut atteindre une source web ni déclencher un workflow de sonde**, donc tout item de scraping est non dispatchable en l'état — les 4 items de sources d'activites-vallauriennes (P2.7a-bis, P2.7a-ter, P2.7b, P2.7c) sont concernés. Constaté le 2026-07-17 : l'issue #16 d'activites-vallauriennes demandait de lancer `probe-sources.yml` et de lire les dumps — infaisable, la session s'est arrêtée après 30 tours (~2 min, 0,62 $) sans rien produire. Nuance : `Bash(node:*)` permet déjà `node -e "fetch(...)"`, donc le réseau est atteignable en pratique par un détour — l'allowlist est incohérente plutôt que fermée. À trancher : ouvrir `WebFetch` (lecture seule, le plus lisible), ou `Bash(gh workflow:*)` + `Bash(gh run:*)` pour les sondes, ou assumer que le scraping se traite en local. DoD : un item de scraping part en dispatch et aboutit à une PR, ou la doc du kit dit noir sur blanc que ces items ne se dispatchent pas. ✅ 2026-07-20 — arbitrage : **`WebFetch` seul** (lecture seule), ajouté aux deux cas de `dispatch.yml` et à `self-heal.yml` ; `gh workflow`/`gh run` écartés (surface trop large sur un repo public pour le seul besoin de sonde). Raison décisive : ce n'est pas une capacité nouvelle — `Bash(node:*)` autorisait déjà `node -e "fetch(...)"`, l'allowlist était incohérente et non fermée. Commit direct sur `main` (repo méta ; une session Actions ne peut pas modifier `.github/workflows/*`). ⚠️ **DoD non close** : reste à constater sur un vrai cas qu'un item de scraping dispatché aboutit à une PR — à faire au prochain dispatch d'un item d'activites-vallauriennes (P2.7a-bis/ter, P2.7b, P2.7c).

DoD : une PR ouverte en draft sort du brouillon sans geste humain, constaté sur un vrai cas. ✅ 2026-07-20 — **DoD volontairement abandonnée**, arbitrage « correctif léger ». Le correctif complet (PAT `FLEET_GH_TOKEN` sur toute la flotte + changement de stubs + bump de kit + re-équipement) a été jugé disproportionné pour un cas qui ne s'est plus produit depuis le 2026-07-15. Ce qui est fait à la place : la découverte que le prétendu filet de sécurité était **décoratif** — `gh pr ready … 2>/dev/null || true` avalait l'échec, le merge partait quand même, se prenait un 405, et `continue-on-error: true` rendait le run **vert** (PR non mergée, personne averti — exactement l'échec silencieux que ces workflows combattent ailleurs). Désormais, `dispatch.yml` et `self-heal.yml` détectent le draft, tentent le `gh pr ready` sans masquer l'erreur, et en cas d'échec **commentent la PR + émettent un `::warning::`** au lieu de merger dans le vide. Une PR draft demande donc toujours un geste humain, mais on le sait. Commit direct sur `main`.
