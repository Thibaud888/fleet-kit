# Backlog

> 1 item = 1 session Claude (issue labellisée `claude` ou session Cloud) = 1 PR.
> Cocher + lien PR quand c'est mergé. `/dispatch` et `/backlog` (claude-ops) lisent ce fichier.
> Priorité optionnelle en tête d'item : `(P1)` urgent · `(P2)` important · `(P3)` un jour.

- [ ] (P1) Le merge auto ne ferme pas l'issue — `dispatch.yml` merge sous `GH_TOKEN: ${{ github.token }}`, or GitHub n'exécute pas les actions en cascade déclenchées par le `GITHUB_TOKEN` par défaut (garde anti-boucle) : le `Closes #N` du corps de PR est donc **ignoré**, et l'issue reste ouverte après merge. Conséquence : l'anti-collision de `/dispatch` et `/backlog` voit des sessions fantômes sur tous les repos, et le Project « Flotte » ne passe jamais en « Mergé » seul. Constaté le 2026-07-17 sur notes-bac-visualisateur#21 et bac-maths#57 (restées ouvertes, mergées par `app/github-actions`) vs Recherche-Emploi#28 (fermée seule, PR mergée à la main avec le compte Thibaud888) — les 3 PR portaient pourtant `Closes #N`. Même racine que l'item `pr-ready` ci-dessous. Correctif le moins cher : `gh issue close` explicite après `gh pr merge` dans l'étape « Merge automatique si CI verte » (0 secret requis) ; alternative : `FLEET_GH_TOKEN` (PAT) au lieu du `github.token`, ce qui réglerait aussi `pr-ready` — à arbitrer ensemble. DoD : une issue dispatchée se ferme seule au merge de sa PR, constaté sur un vrai cas.
- [ ] (P2) L'allowlist des sessions n'a aucune porte de sortie réseau — `dispatch.yml` autorise `Bash(git|gh issue|gh pr|npm|npx|node|python|pip|pytest)`, `Edit/Write/Read/Glob/Grep` : ni `curl`, ni `WebFetch`, ni `gh workflow run`, ni `gh run`. Conséquence : **aucune session dispatchée ne peut atteindre une source web ni déclencher un workflow de sonde**, donc tout item de scraping est non dispatchable en l'état — les 4 items de sources d'activites-vallauriennes (P2.7a-bis, P2.7a-ter, P2.7b, P2.7c) sont concernés. Constaté le 2026-07-17 : l'issue #16 d'activites-vallauriennes demandait de lancer `probe-sources.yml` et de lire les dumps — infaisable, la session s'est arrêtée après 30 tours (~2 min, 0,62 $) sans rien produire. Nuance : `Bash(node:*)` permet déjà `node -e "fetch(...)"`, donc le réseau est atteignable en pratique par un détour — l'allowlist est incohérente plutôt que fermée. À trancher : ouvrir `WebFetch` (lecture seule, le plus lisible), ou `Bash(gh workflow:*)` + `Bash(gh run:*)` pour les sondes, ou assumer que le scraping se traite en local. DoD : un item de scraping part en dispatch et aboutit à une PR, ou la doc du kit dit noir sur blanc que ces items ne se dispatchent pas.
- [ ] (P2) `pr-ready.yml` en échec sur les PR en brouillon — `markPullRequestReadyForReview` est interdit au `GITHUB_TOKEN` (« Resource not accessible by integration »), donc les PR draft (celles ouvertes par les sessions Claude Code web / FleetView) ne sortent pas du brouillon toutes seules et ne peuvent pas être mergées (GitHub renvoie 405) — il faut le faire à la main. Constaté 2 fois sur `activites-vallauriennes` les 2026-07-14 et 2026-07-15 ; **non éprouvé depuis** (tous les runs `pr-ready` depuis sont en `skipped` : aucune PR draft n'est passée — les PR de la flotte naissent prêtes). Piste : token d'app ou PAT (`FLEET_GH_TOKEN`) au lieu du `github.token`. Item remonté du BACKLOG.md d'activites-vallauriennes le 2026-07-17 (il vise le kit, pas le site). DoD : une PR ouverte en draft sort du brouillon sans geste humain, constaté sur un vrai cas.
- [ ] (P2) Déclencher la propagation du kit au bump de VERSION — l'outil existe et est rodé (`claude-ops/scripts/kit-propager.mjs` : 1 PR par repo en retard, merge dès CI verte, idempotent, `--dry-run` ; v1.3.0 propagée sur 10 repos le 2026-07-17), mais son **déclenchement reste manuel** : un bump de VERSION ici laisse la flotte en arrière jusqu'à ce qu'on y pense (l'hygiène du lundi ne fait que signaler la dérive). Reste à décider **où** l'accrocher : workflow de release sur fleet-kit (push sur `main` touchant `VERSION` → lance le script), ou étape de l'hygiène hebdo (rattrapage paresseux, 0 workflow de plus). Prérequis du 1er cas : secret `FLEET_GH_TOKEN` (PAT cross-repo, déjà en place sur claude-ops depuis le 2026-07-15) à poser aussi sur fleet-kit — le `github.token` d'un workflow ne voit que son propre repo. Attention : seul le périmètre « fichiers possédés par le kit » (skills de session, `.kit-version`) est propageable ainsi ; un changement touchant CLAUDE.md/allowlist/workflows demande toujours `/equiper` (merge doux). DoD : un bump de VERSION ouvre les PR d'upgrade sans geste humain, constaté sur un vrai bump.
